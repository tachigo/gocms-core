# GoCMS v2.0 架构方向

> 来源：老板（tachigo）与总助理 2026-03-29 架构讨论
> 状态：方向确认，待 PM 出 PRD

---

## 一、核心理念

**一切皆 Module，Headless CMS 框架。**

GoCMS 不是一个"网站"，而是一个 **CMS 框架**。它的职责是：
1. 管理框架内有哪些内容类型
2. 每种内容类型怎么处理数据
3. 提供标准 API，不管前端

## 二、架构总览

```
GoCMS v2.0 = Module 容器 + 规范接口 + Event 总线

框架层（Core）：
├── Module 注册 & 生命周期管理
├── 依赖解析 & 初始化顺序
├── Event 总线（pub-sub）
├── API 路由统一挂载（GoFrame Bind 模式）
├── OpenAPI / Swagger 自动生成
├── CMD 命令注册
└── 基础中间件管道

一切皆 Module：
├── article    — 内容（文章）
├── page       — 内容（页面）
├── menu       — 内容（树形导航）
├── taxonomy   — 内容（分类/标签体系）
├── media      — 基础设施（文件存储）
├── user       — 基础设施（账号管理）
├── permission — 基础设施（权限，中间件形式）
├── settings   — 基础设施（站点配置）
└── ...        — 未来扩展
```

## 三、Module 规范接口

每个 Module 必须实现框架定义的接口，框架通过接口统一管理全局：

```go
type Module interface {
    // 身份
    Name() string              // 模块标识："article" / "menu" / "user"
    Description() string       // 模块描述

    // 依赖
    Dependencies() []string    // 依赖的其他模块：["user", "media", "taxonomy"]

    // 生命周期
    Init(app *App) error       // 初始化（建表、种子数据、注册事件监听等）

    // 路由
    RegisterRoutes(group *RouterGroup)  // 注册 API 路由（GoFrame Bind 模式）

    // Schema 声明
    Schema() ModuleSchema      // 字段定义、关系、权限需求
}
```

**框架通过规范接口获得的能力：**

| 能力 | 依赖接口 |
|------|---------|
| 自动初始化所有 module（按依赖顺序） | Dependencies() + Init() |
| 统一 API 路由挂载 | RegisterRoutes() |
| 全局权限控制 | Schema() 权限声明 + permission 中间件 |
| OpenAPI / Swagger 自动生成 | RegisterRoutes() 中的 Bind struct |
| 后台 UI 自动渲染（可选） | Schema() 字段类型 + 分组 |
| 健康检查 / 模块状态总览 | 遍历已注册 module |
| CMD 子命令扩展 | module 注册自己的命令 |

## 四、Module 自管原则

每个 Module 在规范内自由发挥，自行管理：

- **字段定义** — Go struct 定义模型（非 YAML），类型安全、IDE 友好、编译检查
- **数据迁移** — module 内部管理，需要时通过 CMD 命令执行
- **业务逻辑** — logic/service 层完全自主
- **自定义 CMD** — 注册自己的命令行子命令（seed、migrate 等）

## 五、Module 目录结构

```
internal/module/
  article/
    logic/        ← 业务逻辑
    service/      ← 对外服务接口
    model/        ← 数据模型（Go struct）
    controller/   ← API 控制器（Request/Response struct）
    cmd/          ← 自定义命令（可选）
  menu/
    logic/
    service/
    model/
    controller/
  user/
    ...
  permission/
    ...
```

## 六、Event 机制

模块间通过 Event 总线解耦通信（发布-订阅模式）：

- **Event（事件）**：通知发生了什么，监听者自行处理（异步、解耦）
- 发布者不需要知道谁在听，监听者不需要知道谁在发

```go
// 文章 module 发布事件
app.Emit("article.published", articleData)

// 缓存 module 监听
app.On("article.published", func(data interface{}) {
    cache.Invalidate("article-list")
})
```

**典型场景：**
- 内容发布 → 缓存清理
- 内容删除前 → 引用检查
- 用户创建 → 初始化默认权限

## 七、权限 Module 特殊性

权限不是内容类型，是行为控制。以 Module 形式注册，运行时作为中间件：

- Init() → 创建角色表、权限表、种子数据
- 注册为全局中间件 → 拦截每个 API 请求做鉴权
- 其他 module 通过 Schema() 声明自己需要的权限

## 八、GoFrame OpenAPI / Swagger 集成

v2.0 每个 module 用 Go struct 定义 Request/Response，天然适配 GoFrame 的 Bind 模式：

```go
type CreateArticleReq struct {
    g.Meta  `path:"/article" method:"post" tags:"文章" summary:"创建文章"`
    Title   string `json:"title" v:"required" dc:"标题"`
    Slug    string `json:"slug" v:"regex:^[a-z0-9-]+$" dc:"URL别名"`
    Body    string `json:"body" v:"required" dc:"正文"`
    Status  string `json:"status" v:"in:draft,published" dc:"状态" d:"draft"`
}
```

GoFrame 自动生成：
- OpenAPI 3.0 文档（`/api.json`）
- Swagger UI（`/swagger`）
- 参数校验 + 默认值 + 字段描述

## 九、与 v1.0 的核心区别

| 维度 | v1.0 | v2.0 |
|------|------|------|
| 核心引擎 | SchemaEngine（大管家） | Module 容器（注册 + 协调） |
| 字段定义 | YAML 配置驱动 | Go struct（module 自管） |
| 数据操作 | map[string]interface{}（动态） | 强类型 struct（静态） |
| 内容类型 | 通用逻辑，差异仅在字段 | 每个 module 有自己的 logic |
| 菜单/页面 | 独立模块，特殊处理 | 也是 module，统一管理 |
| API 文档 | 手写 | OpenAPI 自动生成 |
| 路由模式 | 标准路由 | GoFrame Bind 模式 |
| 迁移 | MigrateEngine | module 内部自管 |
| 模块通信 | 直接调用 | Event 总线解耦 |
