## 🔥 根因分类汇总

本轮 165 个失败 + 38 个错误中，经分析可归为以下几大根因。每个根因标注了影响范围和修复归属（**API Bug** = 后端需修 / **测试适配** = 测试用例需改 / **两者都需要**）。

### 根因 1：文章创建要求 `author_id` 必填（API Bug）⭐ P0 阻塞

- **影响范围：** article 模块 ~50 个用例（含 30 个 fixture 创建 ERROR）
- **现象：** 创建文章返回 `{"code":51,"message":"The AuthorID field is required"}`
- **期望：** API 应从 JWT token 中自动提取 `user_id` 作为 `author_id`，PRD 明确 "created_by/author_id 由后端自动填充"
- **实际：** `author_id` 作为必填验证字段，前端不传则 400
- **根因：** 后端 article 模块验证规则将 `author_id` 设为 required，缺少自动填充逻辑
- **影响用例数：** ~50（含 conftest `created_article` 和 `published_article` fixture 级联失败）

### 根因 2：Menu API 路径与测试不匹配（两者都需要）⭐ P0

- **影响范围：** menu 模块全部 28 个用例
- **现象：**
  - 测试访问 `/api/admin/menus` → 返回 404
  - 测试访问 `/api/admin/menus/main` → 返回 `ID value 'main' must be >= 1`
- **实际 API 路径：**
  - 列表：`GET /api/admin/menu-groups`
  - 树形：`GET /api/admin/menus/{group}/tree`（group 为字符串 slug 如 "main"）
  - 公开 API：`GET /api/menus/{group}/tree`
- **说明：** 测试已知需要适配，但实际 API 也存在问题——不存在的 group 返回 `{"code":0,"data":{"tree":null}}` 而非 404

### 根因 3：Taxonomy vocabulary 使用数字 ID（测试适配 + API 设计确认）

- **影响范围：** taxonomy 模块 ~22 个用例 + conftest `created_term` fixture
- **现象：** 测试用 `tags` / `article_category` 作为 vocabulary ID → 返回 `VocabularyID value 'tags' must be >= 1`
- **实际 API：** vocabulary 使用数字 ID（1=categories, 2=tags），提供 `machine_id` 字段作为字符串标识
- **根因：** 测试用例按 PRD 用字符串 slug，实际 API 路径用数字 ID
- **建议：** API 应同时支持 `machine_id` 路由（如 `/api/admin/taxonomies/tags/terms`），或测试改用数字 ID

### 根因 4：HTTP 错误状态码规范不一致（测试适配）

- **影响范围：** 跨模块约 25 个用例
- **现象：** 资源不存在时测试期望 HTTP 404，实际返回 HTTP 200 + `{"code":50,"message":"xxx不存在"}`
- **实际行为示例：**
  - `GET /api/admin/articles/99999` → HTTP 200 + `{"code":50,"message":"文章不存在"}`
  - `GET /api/admin/articles/abc` → HTTP 200 + `{"code":51,"message":"The ID value 'abc' must be >= 1"}`
- **根因：** API 采用统一 HTTP 200 + 业务 code 的设计模式，测试用例按 RESTful 标准期望 4xx 状态码
- **判定：** API 设计选择，测试应适配（改为 `expect_status=200, expect_code=50`）

### 根因 5：conftest `expect_code` 默认值导致误判（测试适配）

- **影响范围：** 跨模块约 15 个用例（401/403 场景）
- **现象：** 测试 `expect_status=401` 时，`expect_code` 默认为 0，但实际 code 为 401 → 断言失败
- **根因：** `api_request()` 中 `expect_code=0` 是默认值，当测试 401/403 场景时应传 `expect_code=None` 或对应值
- **受影响用例：** `test_create_no_auth`、`test_no_auth`、`test_non_admin_access_roles` 等

### 根因 6：Session-scoped token 被 logout 测试失效（测试设计缺陷）

- **影响范围：** user 模块约 30 个后续用例级联 401
- **现象：** `test_logout_success` 和 `test_logout_token_invalidated` 注销了 session 级别的 admin_token，导致同文件后续所有用例 401
- **根因：** logout 测试应使用独立获取的 token，不能消耗 session-scoped fixture
- **判定：** 纯测试框架问题

### 根因 7：RBAC 角色/权限配置与 PRD 不符（API Bug）⭐ P0

- **影响范围：** permission 模块 ~15 个、article/page 各 ~5 个
- **现象：**
  - 系统只有 `admin` / `editor` / `viewer` 三个角色，**缺少 `author` 角色**
  - `editor` 无法读取文章列表（返回 403）
  - `author`（实为 viewer？）无法读取文章/页面/taxonomy
- **期望：** PRD 定义 editor 应有文章全权限（read:all / write:all），author 应有 read:own / write:own
- **根因分析：**
  - `author` 角色未初始化到系统中
  - `editor` 角色权限配置不完整，缺少 article/page/taxonomy/media 的 read/write 权限

### 根因 8：响应字段缺失/命名不匹配（API Bug + 测试适配）

| 模块 | 期望字段 | 实际字段 | 性质 |
|------|---------|---------|------|
| framework | `/health` 包含 `data.modules` | 直接返回 `{modules, status, version}`，无 `data` 包装 | 测试适配 |
| user | login 返回 `user.roles` | 无 `roles` 字段 | 需确认 PRD |
| user | JWT payload 含 `roles` | 仅含 `user_id, username, iss, exp, iat` | 需确认 PRD |
| media | upload 返回 `mime_type, size, filename` | 仅返回 `{id, url}` | **API Bug** |
| media | `styles`（缩略图信息） | 无 `styles` 字段 | 需确认 PRD |
| media | folder 含 `parent_id` | 无 `parent_id` 字段 | **API Bug** |
| taxonomy | `hierarchical` | `hierarchy` | 测试适配 |
| page | `seo_title`, `seo_description` 等 | 无 SEO 字段 | **API Bug** |
| page | 默认 `template = "default"` | 实际为空字符串 `""` | **API Bug** |

## 🛡️ 安全问题（P0）

### SEC-01: XSS 未过滤 — Page body 存储型 XSS

- **模块：** page
- **用例：** `test_page.py::TestPageAdminP0::test_xss_body`
- **现象：** 创建 page 时 body 含 `<script>alert(1)</script><p>safe</p>`，存储后原样返回
- **期望：** `<script>` 标签应被过滤或转义
- **实际：** body 原样存储和返回 `<script>alert(1)</script><p>safe</p>`
- **风险等级：** 🔴 高 — 存储型 XSS，攻击者可注入恶意脚本
- **建议：** 后端存储时使用 HTML sanitizer 过滤危险标签

### SEC-02: 危险文件类型上传未拦截

- **模块：** media
- **用例：** `test_media.py::TestMediaUploadP0::test_upload_invalid_type`
- **现象：** 上传 `.exe` 文件返回 `{"code":0}`，上传成功
- **期望：** 应拒绝可执行文件（.exe/.bat/.sh 等）
- **实际：** 上传成功，media list 中可见 `mime_type: application/x-msdownload`
- **风险等级：** 🔴 高 — 可被利用上传恶意可执行文件
- **建议：** 增加文件类型白名单校验（只允许 image/video/audio/document 等安全类型）

### SEC-03: 禁用用户仍可登录

- **模块：** user
- **用例：** `test_user.py::TestLoginP0::test_login_disabled_user`
- **现象：** 被标记为 disabled 的用户 login 返回 `code:0`，登录成功
- **期望：** disabled 用户应返回 `code != 0`，拒绝登录
- **风险等级：** 🟡 中 — 已禁用的账号仍可访问系统

## 🐛 功能缺陷（P0/P1）

### BUG-01: 页面发布状态不生效 ⭐ P0

- **模块：** page
- **用例：** `test_page.py::TestPageAdminP0::test_publish_page`
- **现象：** PUT 更新 page `status = "published"` 后，重新 GET 查询仍为 `"draft"`
- **期望：** status 应更新为 `"published"`
- **影响：** 级联导致公开 API 获取已发布页面返回"页面不存在"（public API 只返回 published 状态）
- **关联失败用例：** `test_public_detail_by_id`、`test_public_detail_by_slug`、`test_public_template_field`、`test_public_seo`、`test_public_cover_image`（共 5 个）

### BUG-02: 文章 author_id 未自动填充 ⭐ P0

- **详见根因 1**
- **阻塞：** article 模块几乎所有写操作均失败

### BUG-03: Upload 响应缺少元数据

- **模块：** media
- **现象：** `POST /api/admin/media/upload` 成功后仅返回 `{"id": 9, "url": "/uploads/..."}`
- **期望：** 应返回完整信息：`mime_type, size, filename, uploaded_by, created_at` 等
- **影响用例：** `test_upload_png`、`test_upload_jpeg`、`test_upload_gif`（均 KeyError: 'mime_type'）

### BUG-04: Page 不支持 SEO 字段

- **模块：** page
- **现象：** 创建 page 传入 `seo_title`, `seo_description`, `seo_keywords` 后，详情中无这些字段
- **期望：** PRD 要求支持 SEO 元数据

### BUG-05: Page 默认 template 为空

- **模块：** page
- **现象：** 创建 page 不传 template 时，`template = ""`（空字符串）
- **期望：** 应默认为 `"default"`

### BUG-06: 大文件上传由 nginx 直接 413 拒绝

- **模块：** media
- **现象：** 上传 11MB 文件返回 HTTP 413（nginx），非应用层响应
- **建议：** 应在应用层限制并返回友好的业务错误码
