# GoCMS 微内核架构升级 - 产品需求文档 (PRD)

> 版本：v1.0 | 日期：2026-03-31 | 作者：阿产

---

## 1. 背景与产品价值

### 1.1 当前痛点

| 痛点 | 影响 |
|------|------|
| 每个新项目都要从零搭建后台 | 重复造轮子，交付周期长（2-4周） |
| 单体应用臃肿，功能耦合 | 改一处动全身，风险高 |
| 多系统用户数据孤岛 | 用户需要记住多套账号密码 |
| 模块无法复用 | 相似功能在不同项目重复开发 |
| 升级困难 | 底层框架升级需改动所有项目 |

### 1.2 产品价值

**一句话价值主张：**
> 让后台系统开发像搭积木一样简单 —— 选择需要的模块，一键组装，一周交付。

| 价值维度 | 具体收益 |
|----------|----------|
| **快速交付** | 新项目从 2-4 周缩短至 3-5 天 |
| **成本降低** | 复用现有模块，减少 60%+ 重复开发 |
| **质量提升** | 成熟模块经多项目验证，Bug 更少 |
| **灵活扩展** | 按需组装，不用的功能零负担 |
| **统一体验** | 多系统共享用户中心，单点登录 |

## 2. 目标用户

### 2.1 用户画像

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| **后台开发工程师** | 需要快速交付管理后台的开发者 | 少写重复代码，专注业务逻辑 |
| **技术负责人/架构师** | 负责技术选型和架构设计 | 系统可维护、可扩展、低风险 |
| **产品经理** | 需要快速验证产品想法 | 快速上线 MVP，低成本试错 |
| **运维工程师** | 负责系统部署和维护 | 部署简单，配置集中管理 |

### 2.2 用户分层

```
┌─────────────────────────────────────────────────────────┐
│  第一层：平台开发者（我们团队）                              │
│  - 开发并维护模块仓库                                      │
│  - 提供脚手架工具和文档                                    │
├─────────────────────────────────────────────────────────┤
│  第二层：应用开发者（使用方）                               │
│  - 使用脚手架创建新项目                                    │
│  - 按需安装模块组装系统                                    │
│  - 编写业务专属代码                                        │
├─────────────────────────────────────────────────────────┤
│  第三层：最终用户（管理员/运营）                            │
│  - 使用组装好的系统进行日常管理                            │
└─────────────────────────────────────────────────────────┘
```

## 3. 功能清单

### 3.1 功能总览

| 模块 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| **脚手架工具** | 初始化新项目 | P0 | 一键生成项目骨架 |
| | 安装模块 | P0 | 从模块库安装指定模块 |
| | 卸载模块 | P0 | 移除已安装模块 |
| | 查看已安装模块 | P1 | 列表展示当前项目模块 |
| **模块仓库** | 用户认证模块 | P0 | 登录/注册/权限管理 |
| | 内容管理模块 | P0 | 文章/页面/分类管理 |
| | 媒体管理模块 | P0 | 文件上传/图片处理 |
| | 系统设置模块 | P0 | 站点配置/菜单管理 |
| | 支付模块 | P1 | 订单/支付/退款 |
| | 消息通知模块 | P1 | 短信/邮件/站内信 |
| **配置中心** | 模块配置管理 | P0 | YAML 配置覆盖默认行为 |
| | 环境变量支持 | P0 | 敏感配置走环境变量 |
| | 配置热加载 | P2 | 不重启更新配置 |
| **用户中心** | 统一身份认证 | P0 | SSO 单点登录 |
| | 权限管理 | P0 | RBAC 角色权限 |
| | 用户资料管理 | P0 | 个人信息/密码修改 |

### 3.2 优先级定义

- **P0（必须有）**：核心功能，没有则产品不可用
- **P1（应该有）**：重要功能，显著提升体验
- **P2（可以有）**：增值功能，后续迭代添加

## 4. 用户场景与流程

### 场景一：快速搭建新项目（核心场景）

**用户：** 后台开发工程师小王

**背景：** 公司需要一个新的项目管理系统，要求 1 周内上线。

**流程：**

```
1. 初始化项目（5分钟）
   └─ 执行：./tool.sh init my-project
   └─ 输出：生成项目骨架，包含基础配置

2. 安装所需模块（10分钟）
   ├─ ./tool.sh install user        # 用户认证
   ├─ ./tool.sh install permission  # 权限管理
   ├─ ./tool.sh install media       # 文件上传
   └─ ./tool.sh install article     # 内容管理

3. 调整配置（15分钟）
   └─ 编辑 config/modules/user.yaml
      ├─ 关闭注册功能（内部系统）
      ├─ 配置企业邮箱域名限制
      └─ 设置密码复杂度要求

4. 启动验证（5分钟）
   └─ go run main.go
   └─ 访问 http://localhost:8080
   └─ 使用默认管理员账号登录验证

5. 开发业务代码（剩余时间）
   └─ 专注编写项目管理专属功能
   └─ 无需关心用户/权限等基础功能
```

**预期结果：** 3 小时内完成基础系统搭建，剩余 4 天专注业务开发。

---

### 场景二：已有项目添加新功能

**用户：** 技术负责人李总

**背景：** 现有 CMS 系统需要增加支付功能，传统方式需开发 2 周。

**流程：**

```
1. 评估模块（10分钟）
   └─ 查看模块仓库，发现已有 payment 模块
   └─ 阅读文档，确认满足需求

2. 安装模块（5分钟）
   └─ ./tool.sh install payment
   └─ 自动创建软链接、更新配置

3. 配置支付参数（20分钟）
   └─ 编辑 config/modules/payment.yaml
      ├─ 配置支付宝/微信商户号
      ├─ 设置回调地址
      └─ 配置订单超时时间

4. 权限配置（10分钟）
   └─ 在后台为运营角色分配支付管理权限

5. 联调测试（1天）
   └─ 沙箱环境测试支付流程
   └─ 验证订单状态流转
```

**预期结果：** 1 天完成支付功能集成，而非 2 周开发。

---

### 场景三：多系统统一用户中心

**用户：** 架构师老张

**背景：** 公司有 5 个内部系统，各自独立账号，用户体验差，运维成本高。

**流程：**

```
1. 搭建统一用户中心（1天）
   ├─ ./tool.sh init sso-center
   ├─ ./tool.sh install user
   ├─ ./tool.sh install permission
   └─ 部署为独立服务（sso.company.com）

2. 改造现有系统（每系统 0.5 天）
   ├─ 安装 SSO 客户端模块
   ├─ 配置用户中心地址
   └─ 移除原有登录逻辑

3. 用户迁移（1天）
   ├─ 导出各系统用户数据
   ├─ 导入用户中心
   └─ 发送密码重置邮件

4. 验证上线（1天）
   ├─ 测试跨系统登录
   ├─ 验证权限同步
   └─ 灰度发布
```

**预期结果：** 1 周完成 5 个系统 SSO 改造，用户只需一套账号。

---

### 场景四：模块版本升级

**用户：** 运维工程师小赵

**背景：** user 模块发现安全漏洞，需要紧急升级。

**流程：**

```
1. 查看当前版本
   └─ ./tool.sh list
   └─ 显示：user@v1.0.0

2. 拉取新版本
   └─ cd ../gocms-modules/user
   └─ git pull origin main
   └─ 更新到 v1.0.1（安全补丁）

3. 重启服务
   └─ 重新编译部署
   └─ 验证登录功能正常

4. 回滚准备
   └─ 保留 v1.0.0 标签
   └─ 如有问题可快速回滚
```

**预期结果：** 10 分钟完成安全补丁升级，零代码改动。

## 5. 验收标准

### 5.1 功能验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 项目初始化 | 执行 `./tool.sh init` 后，生成可编译运行的项目骨架 | 手动执行验证 |
| 模块安装 | 执行 `./tool.sh install user` 后，系统具备完整用户功能 | 功能测试 |
| 模块卸载 | 执行 `./tool.sh uninstall user` 后，用户功能完全移除 | 功能测试 |
| 配置覆盖 | 修改 `config/modules/user.yaml` 后，模块行为按配置改变 | 配置测试 |
| 环境变量 | 设置 `GOCMS_USER_JWT_SECRET` 后，配置被覆盖 | 配置测试 |
| 依赖解析 | 安装 article 模块时，自动处理 taxonomy 依赖 | 依赖测试 |

### 5.2 性能验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 启动时间 | 8 个模块的系统启动时间 < 3 秒 | 计时测试 |
| 模块安装 | 单个模块安装耗时 < 30 秒 | 计时测试 |
| 内存占用 | 空载系统内存占用 < 100MB | 监控测试 |

### 5.3 用户体验验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 新手友好 | 新用户按文档可在 1 小时内完成首个项目搭建 | 用户测试 |
| 错误提示 | 操作失败时，提示信息清晰可理解 | 人工检查 |
| 文档完整 | 每个功能点都有对应的使用文档 | 文档审查 |

## 6. 非功能需求

### 6.1 兼容性

- 模块向后兼容：新版本模块不得破坏已有功能
- 配置兼容：配置项变更需保留旧配置兼容期

### 6.2 安全性

- 敏感配置（密钥、密码）必须走环境变量
- 模块安装来源需可验证（签名/校验和）

### 6.3 可维护性

- 模块代码独立，互不影响
- 单个模块升级不影响其他模块
